Lab 1

Runbook Authoring

Pillar: Operational Excellence

Objective

Write a runbook clear enough that another engineer — woken at 3 AM, never having seen the system — could execute it successfully.

Then identify which steps you would automate first.

Scenario

  • Customer-facing API on EC2 ASG behind an ALB
  • Aurora MySQL backend
  • Alert just fired:
    api-prod / 5xx error rate > 5% for 10 min
  • Usually caused by Aurora connection pool exhaustion from a stuck app thread

Your Runbook Must Include

  1. Title & symptom
  2. Pre-checks — confirm this is the right runbook
  3. Mitigation steps — ordered, specific, with expected outcome
  4. Verification — how you know it worked
  5. Rollback
  6. Escalation
  7. Post-incident — what to capture for the retro

Constraints

  • No step may say "investigate" or "check things" — be concrete
  • Every command should be copy-pasteable
  • Maximum one page

Then…

Mark each step:

  • [A] — automate this next quarter
  • [H] — keep human-driven

Be ready to defend your choices.

Discussion

  • Which step was hardest to write concretely? Why?
  • What CloudWatch dashboard would make this runbook faster?
  • If this fires twice a month, what should change in the system — not just the runbook?
