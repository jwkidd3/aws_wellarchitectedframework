Lab 2

IAM Least Privilege Review

Pillar: Security

Objective

Read a sample IAM policy attached to an application role, identify least-privilege violations, and rewrite it so the application can do its job — and nothing else.

Least privilege = grant only the permissions that are actually needed.

Terms You'll See

  • IAM (Identity and Access Management) — AWS's service for controlling who can do what
  • IAM policy — a JSON document listing what actions are allowed (or denied) on what resources
  • IAM role — an identity that workloads (e.g., an EC2 instance) can assume, instead of storing credentials in the app
  • ARN (Amazon Resource Name) — a globally-unique ID for an AWS resource, e.g., arn:aws:s3:::my-bucket
  • SQS (Simple Queue Service) — AWS's managed message queue for decoupling components
  • DynamoDB — AWS's managed NoSQL key-value database
  • CloudWatch Logs — log storage and search; apps write to a log group
  • PassRole — an IAM permission that lets a principal hand a role to a service (a classic privilege-escalation hole if granted broadly)

Scenario — app-orders-processor

You're reviewing the IAM role attached to an application that processes customer orders.

  • Reads order events from one SQS queue:
    arn:aws:sqs:us-east-1:111122223333:orders-incoming
  • Writes to one DynamoDB table:
    arn:aws:dynamodb:us-east-1:111122223333:table/orders
  • Writes structured logs to its own CloudWatch log group
  • Runs on EC2 in account 111122223333, region us-east-1

The Current Policy

{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "AppAccess",
      "Effect": "Allow",
      "Action": "*",
      "Resource": "*"
    },
    {
      "Sid": "ExtraStuff",
      "Effect": "Allow",
      "Action": [
        "s3:*", "iam:PassRole",
        "dynamodb:*", "sqs:*", "logs:*"
      ],
      "Resource": "*"
    }
  ]
}

"*" in Action means "every possible action." "*" in Resource means "every resource in the account."

Tasks

  1. List at least 5 violations. Consider:
    • Action wildcards and resource wildcards
    • Privileges the workload does not need at all
    • Privilege-escalation risks (like iam:PassRole)
    • Cross-region or cross-account exposure
  2. Rewrite the policy — scoped by action and resource ARN
  3. Bonus: add a Condition that restricts the role to us-east-1

Hints

  • DynamoDB actions you likely need: GetItem, PutItem, UpdateItem, BatchWriteItem
  • SQS actions you likely need: ReceiveMessage, DeleteMessage, GetQueueAttributes
  • CloudWatch Logs: CreateLogStream, PutLogEvents — scope to the app's log group ARN
  • Region condition key: aws:RequestedRegion

Sample Solution

{
  "Statement": [
    {
      "Sid": "ReadOrderQueue",
      "Effect": "Allow",
      "Action": ["sqs:ReceiveMessage","sqs:DeleteMessage","sqs:GetQueueAttributes"],
      "Resource": "arn:aws:sqs:us-east-1:111122223333:orders-incoming"
    },
    {
      "Sid": "WriteOrdersTable",
      "Effect": "Allow",
      "Action": ["dynamodb:GetItem","dynamodb:PutItem","dynamodb:UpdateItem","dynamodb:BatchWriteItem"],
      "Resource": "arn:aws:dynamodb:us-east-1:111122223333:table/orders"
    },
    {
      "Sid": "WriteAppLogs",
      "Effect": "Allow",
      "Action": ["logs:CreateLogStream","logs:PutLogEvents"],
      "Resource": "arn:aws:logs:us-east-1:111122223333:log-group:/app/orders-processor:*"
    }
  ]
}

Console Exploration (read-only)

Navigate to these in the AWS Console. Do not create, edit, or save anything.

  1. Search for IAM in the top search bar → Policies
  2. Filter by type AWS managed. Open AdministratorAccess and view the JSON tab — notice the wildcards
  3. Open ReadOnlyAccess and scroll through — notice how specific the actions are
  4. Back in IAM, open Access Analyzer in the left nav. Browse any findings that exist

Discussion

  • How would you detect a policy like the original before it reaches production?
  • Which AWS services help enforce least-privilege?
    IAM Access Analyzer (flags over-broad access) · SCPs (account-wide deny rules) · Config rules
