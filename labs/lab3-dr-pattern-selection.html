Lab 3

DR Pattern Selection

Pillar: Reliability

Objective

For each of three workloads, choose a DR (Disaster Recovery) pattern, justify it against the required RTO/RPO and budget, and identify what you would test to prove it works.

Terms You'll See

  • RTO (Recovery Time Objective) — how long the system can be down before it must be back
  • RPO (Recovery Point Objective) — how much data you can afford to lose (e.g., "last 5 minutes")
  • Region — a geographic AWS location (e.g., us-east-1)
  • AWS Backup — a central service that schedules and manages backups across many AWS services
  • CRR (Cross-Region Replication) — automatic copying of data (e.g., S3 objects) to another region
  • Aurora Global Database — an Aurora DB spanning multiple regions with <1 second replication
  • DynamoDB Global Tables — multi-region DynamoDB with automatic two-way replication
  • Route 53 ARC (Application Recovery Controller) — a DNS-based service for controlled failover between regions

Pilot Light = minimal copy of the system always running elsewhere. Warm Standby = full copy, scaled down, ready to scale up. Active/Active = both sides handle live traffic.

Workload A — Internal HR Reporting

  • ~50 internal users, business hours only
  • Aurora MySQL backend, ~20 GB of data
  • Outage tolerable for a full workday; data loss must be < 24 hours
  • Budget: minimal (funded by internal IT chargeback)

Workload B — E-Commerce Storefront

  • ~5,000 concurrent shoppers at peak, 24×7
  • DynamoDB + Lambda + CloudFront (AWS's CDN)
  • Every minute of downtime = lost revenue and brand damage
  • Compliance: data must stay in-region or in approved regions
  • Budget: significant — revenue protection justifies it

Workload C — Real-Time Trading Platform

  • Tier-0 (most critical tier) financial workload
  • Sub-second order routing
  • Any data loss is unacceptable
  • Any meaningful downtime triggers regulatory reporting
  • Budget: essentially open

For Each Workload

  1. Pattern choice — pick one pattern and justify in 2–3 sentences
  2. Target RTO & RPO — be specific
  3. AWS services — what you'd actually use (AWS Backup, CRR, Route 53 ARC, Aurora Global Database, S3 CRR, DynamoDB Global Tables…)
  4. Game-day test — one failure scenario you'd rehearse
  5. Top risk — the thing most likely to break the plan

Console Exploration (read-only)

Navigate only — do not create, edit, or delete anything.

  1. Search for AWS Backup → Backup plans. Browse any existing plans and their rules.
  2. Open Backup vaults in the left nav and look at what's stored.
  3. Search for Route 53 → Hosted zones. Notice how DNS records control traffic routing — the foundation of DR failover.
