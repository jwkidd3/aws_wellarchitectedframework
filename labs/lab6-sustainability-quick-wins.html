Lab 6

Sustainability Quick Wins

Pillar: Sustainability

Objective

Review a small reference architecture, list sustainability improvements, and show how each one also improves at least one other pillar (usually Cost or Performance).

Terms You'll See

  • Instance family — a group of EC2 types optimized for a purpose (m5 = general, c5 = compute, r5 = memory, db.r5 = DB memory-optimized)
  • Auto Scaling — automatically adjust how many servers are running based on load
  • S3 Standard / Intelligent-Tiering / Glacier — storage classes from hot to cold; colder = cheaper but slower to retrieve
  • Lifecycle policy — automated rule like "move to Glacier after 30 days, delete after 1 year"
  • CDN (Content Delivery Network) — caches content at edge locations near users. AWS's CDN is CloudFront.
  • Graviton — AWS-designed ARM CPUs; more work per watt, usually cheaper
  • ALB (Application Load Balancer) — distributes HTTP requests across servers

The Architecture (B2B SaaS Analytics)

"B2B SaaS" = Business-to-Business Software as a Service — a cloud application sold to other companies.

  • Region: us-east-1 only (chosen years ago, no specific reason)
  • Web tier: 12 × m5.xlarge EC2, fixed 24×7, average CPU 12%
  • API tier: 8 × c5.2xlarge EC2, fixed 24×7, average CPU 18%
  • Workers: 6 × m5.2xlarge EC2, nightly batch ~2 hours, idle 22 hours

The Architecture (cont.)

  • Database: Aurora MySQL, 2 × db.r5.4xlarge (writer + reader) + identical staging clone running 24×7
  • Storage: 45 TB in S3 Standard, ~70% not accessed in 6+ months, no lifecycle policies
  • Logging: CloudWatch Logs with no retention (keeps everything forever)
  • CDN: None — every request hits the ALB, including the static dashboard JavaScript bundle

Tasks

  1. List 6–8 sustainability improvements. For each:
    • What you'd change
    • Which other pillar benefits (Cost / Performance / Reliability / OpEx)
    • Effort: Small / Medium / Large
  2. Pick the top 3 to do this quarter — defend the ordering
  3. Identify one improvement that would worsen another pillar if done carelessly, and how you'd avoid the trap

Hints — Look For

  • Idle capacity and oversized instances
  • Missing auto-scaling on tiers with variable load
  • Missing storage tiering or lifecycle
  • Missing CDN for static content
  • Missing log retention policies
  • Region choice for non-latency-sensitive work
  • Older instance generations — could Graviton help?
