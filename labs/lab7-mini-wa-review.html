Capstone Lab · 45 min

Lab 7

Mini Well-Architected Review

All six pillars

Objective

Run a lightweight Well-Architected Review against a sample workload, assign risk levels across all six pillars, and produce a prioritized remediation roadmap.

Work in pairs or small groups. The point is the conversation the framework forces — not a perfect score.

Terms You'll See

  • Workload — the collection of components that delivers some business value
  • Single-AZ — running in only one Availability Zone, so an AZ failure takes the workload down
  • AdministratorAccess — an AWS-managed IAM policy that grants full permissions; almost always too broad
  • Snapshot — a point-in-time backup of a database or disk volume
  • Tier-0 — the highest criticality tier; failure has severe business or regulatory consequences
  • Jenkins — a popular open-source CI/CD (continuous integration/deployment) tool
  • Savings Plan — commitment to a spend rate for 1 or 3 years for a discount

The Workload — "OrderFlow"

B2B order-management SaaS, ~3 years old. The team inherited it last quarter.

  • Single AWS account, us-east-1 only
  • ALB → 6 × m5.large EC2 (fixed, no Auto Scaling), running Node.js
  • Aurora MySQL, 1 writer + 1 reader, single AZ
  • S3 bucket for customer file uploads (no lifecycle rules)
  • Jenkins box on EC2 deploys via SSH and shell scripts

OrderFlow (cont.)

  • CloudWatch metrics on by default; no custom dashboards
  • IAM: 14 users (humans + a couple of CI bots); several with AdministratorAccess
  • Backups: nightly Aurora snapshot, 7-day retention, never tested
  • One runbook in a Google Doc, last updated 14 months ago
  • Cost: ~$11k/month, no Savings Plans, no tagging policy
  • No CDN — static assets served from EC2
  • Customers report intermittent slow page loads at peak

For Each Pillar

  1. Note 2–3 findings — specific issues you'd flag
  2. Assign a risk: High / Medium / None
  3. Pick the single highest-impact fix for that pillar

Six pillars: Operational Excellence · Security · Reliability · Performance · Cost · Sustainability

Then Across All Pillars

  1. Roadmap — top 5 items overall, in order, with rough effort (S/M/L)
  2. Cross-pillar wins — fixes that help two or more pillars
  3. Trade-offs — where fixing one pillar hurts another, and how you'd negotiate

Deliverable — One-Page Summary

A one-page summary you'd present to the engineering manager:

  • The 3 things you'd do this sprint
  • The 2 things you'd do this quarter
  • The 1 thing leadership needs to know about now

Console Exploration (read-only)

Navigate only — do not define a workload or save anything.

  1. Search for Well-Architected Tool in the console.
  2. Click Lenses in the left nav — browse the lens catalog (Serverless, SaaS, Machine Learning, etc.). Each is a pillar add-on for a specific workload type.
  3. Back on the dashboard, browse any existing workloads. Open one and look at the pillar question structure — this is exactly the conversation you just had on paper.

Debrief

  • Where did your team disagree most — priority, or facts?
  • How would answers change if OrderFlow were Tier-0 financial software? Or an internal tool for 20 users?
  • Which pillar was hardest to assess from the description alone? What would you ask the team?
