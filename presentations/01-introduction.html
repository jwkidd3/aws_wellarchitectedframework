Module 1 — Introduction

What & Why

  • AWS = Amazon Web Services, the cloud platform
  • The Well-Architected Framework is AWS's set of best practices for building cloud systems, distilled from thousands of customer reviews
  • It gives engineering, security, and finance teams a shared vocabulary for trade-offs
  • Catches risk before it becomes an incident

3. Reliability

4. Performance Efficiency

5. Cost Optimization
Avoid waste
6. Sustainability
Reduce footprint

One pillar per module. Each pillar has a hands-on lab.

The Shared Responsibility Model

A foundational AWS concept that runs through every pillar.

  • AWS is responsible for security of the cloud: data centers, hardware, the hypervisor, the physical network, and the underlying services
  • You are responsible for security in the cloud: your data, your code, your IAM policies, your network config, your patches
  • The line shifts by service: with EC2 you patch the OS; with Lambda AWS does; with S3 AWS runs the storage but you own access control
  • When something goes wrong, knowing which side of the line the problem sits on is the first question

Common AWS Services You'll See Today

  • EC2 (Elastic Compute Cloud) — virtual servers you rent by the hour/second
  • S3 (Simple Storage Service) — object storage for files, backups, logs
  • RDS (Relational Database Service) — managed databases (MySQL, PostgreSQL, etc.)
  • Aurora — AWS's high-performance MySQL/PostgreSQL-compatible database
  • Lambda — run code without managing servers ("serverless")
  • VPC (Virtual Private Cloud) — your own private network inside AWS
  • IAM (Identity and Access Management) — who can do what
  • CloudWatch — metrics, logs, and alarms

The Well-Architected Tool

A workload = the collection of components that delivers some business value (an app, a service, a pipeline).

  • A free service in the AWS Console
  • You define a workload and then answer a questionnaire, pillar by pillar
  • Each answer maps to one or more best practices; unmet practices become findings
  • Findings are scored High Risk / Medium Risk / None and linked to an improvement plan (AWS docs, services, patterns)
  • You can save a milestone (a snapshot) and re-review later to track progress

A Typical Review Flow

  1. Architect and pillar owners meet for 60–90 min
  2. Walk through each pillar's questions together, answer honestly
  3. Tool produces a risk report at the end
  4. Group findings into themes, assign owners, set target dates
  5. Add remediation items to the normal engineering backlog
  6. Re-run the review quarterly or at major milestones

This is what you'll practice in the capstone lab.
