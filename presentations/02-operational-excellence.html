Pillar 1

Operational Excellence

How you run, monitor, and improve systems over time.

Design Principles

  • Operations as code — script everything you'd otherwise do by hand
  • Small, reversible changes — deploy often, limit blast radius
  • Anticipate failure — pre-mortems & game days (practice incidents)
  • Learn from every event — every incident produces an action item
  • Refine procedures frequently — runbooks decay if untouched

SLIs, SLOs, and SLAs

The vocabulary of "how healthy is healthy enough?"

  • SLI (Service Level Indicator) — a specific measurement: "request success rate", "p99 latency"
  • SLO (Service Level Objective) — the target for an SLI: "99.9% success over 30 days"
  • SLA (Service Level Agreement) — the SLO in a contract, with consequences if missed
  • Error budget = 100% minus SLO. Under budget: ship features. Over budget: slow down, invest in reliability.
  • Alarms should fire on burning through the error budget, not on raw metric thresholds

Observability Stack

Observability = the ability to understand what a system is doing from the outside, without having to ship new code.

  • Metrics — numbers over time (CPU at 42%, 123 requests/sec). Cheap to store, fast to query.
  • Logs — text events ("user 42 logged in"). Richer than metrics, more expensive to store.
  • Traces — the full path of one request across every service it touched, with timing for each hop.

AWS Observability Services

  • CloudWatch — metrics, logs, alarms, and dashboards in one service
  • X-Ray — distributed tracing: pinpoint the slow span in a multi-service request
  • CloudTrail — audit log of every API call made in your account (who did what, when)

Three questions these services answer: Is it healthy? Where's the latency? Who changed it?

Runbooks & Playbooks

  • Runbook — a written procedure for a known situation ("when X fires, do A-B-C")
  • Playbook — investigative steps for an unknown situation ("something's wrong, here's how to look")
  • Both should live in version control next to the code
  • Anything you do twice by hand should be scripted

Incident → Runbook → Automation

The maturity cycle every ops team should be climbing.

  1. Incident happens — someone figures it out under pressure
  2. Retro — what did we do, what should we have done
  3. Runbook written — next time it takes 10 minutes, not 2 hours
  4. Alert auto-links to the runbook — on-call doesn't hunt for it
  5. Automation — the runbook becomes a script, then a self-healing system
  6. Prevention — fix the root cause so the alert never fires again

Infrastructure as Code (IaC)

IaC = define your cloud resources in text files that are version-controlled and reviewed like application code.

  • CloudFormation — AWS's native IaC; you write YAML or JSON templates and AWS creates the resources
  • CDK (Cloud Development Kit) — define infrastructure in TypeScript/Python/Java; compiles down to CloudFormation
  • Terraform — popular third-party IaC tool, multi-cloud
  • Benefits: repeatable, peer-reviewable, drift-detectable, disaster-recoverable
  • Changes made manually in the console are "operational debt"

CI/CD in One Slide

CI/CD = Continuous Integration / Continuous Delivery — automated build, test, and deploy.

  • CI — every commit builds, runs tests, and produces an artifact
  • CD — that artifact is automatically promoted through environments (dev → staging → prod)
  • AWS services: CodePipeline, CodeBuild, CodeDeploy — or use GitHub Actions, GitLab, Jenkins
  • Deployments should be one-click, observable, and reversible in under 5 minutes
  • "Manual deployment" is an antipattern, not a step

Game Days

  • Scheduled exercises where the team deliberately breaks something and practices the response
  • Classic targets: kill a random instance, block an AZ, exhaust a connection pool
  • Purpose: find the gaps in runbooks, monitoring, and team knowledge before a real incident
  • Tools: AWS Fault Injection Simulator (FIS), Netflix's Chaos Monkey
  • Run regularly — a game day once a year is theater, not practice
