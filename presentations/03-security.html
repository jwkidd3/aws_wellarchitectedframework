Pillar 2

Security

Protect data, systems, and who-can-do-what.

Defense in Depth

No single control is enough. Layer them so an attacker has to defeat several.

  • Identity layer — strong authentication, MFA, least privilege
  • Network layer — VPCs, security groups, WAF, private subnets
  • Application layer — input validation, dependency scanning, secrets management
  • Data layer — encryption at rest and in transit, key rotation, backups
  • Detection layer — logs, alerts, threat detection

If one layer fails, the next one should slow the attacker down.

Authentication vs. Authorization

Two different questions people often conflate.

  • Authentication = who are you? — proving identity (password, MFA, federated login)
  • Authorization = what can you do? — what that identity is allowed to do once inside
  • A user can be authenticated (logged in) and still be denied an action (not authorized)
  • IAM covers both — but most IAM policy work is authorization

Identity & Access Management (IAM)

IAM is the AWS service that controls who (or what) can make which API calls on which resources.

  • Users for humans, roles for workloads — never share long-lived credentials
  • Least privilege — grant only the permissions actually needed, scoped to specific resources
  • Prefer federation (log in via your company's identity provider) over creating IAM users
  • IAM Access Analyzer — an AWS tool that flags policies granting access outside your account

MFA and Root Account Protection

  • MFA (Multi-Factor Authentication) — password plus a second factor (phone app, hardware key)
  • Every human account should have MFA enabled — no exceptions
  • The root account is the original all-powerful account; lock it away:
    • Hardware MFA
    • No access keys
    • Used only for billing or to recover access
  • Do daily work with IAM users or federated identities — not root

Anatomy of an IAM Policy

Four things a policy statement specifies:

  • Effect — Allow or Deny
  • Action — which API calls (s3:GetObject)
  • Resource — which ARN(s) the actions apply to
  • Condition — optional constraints (region, source IP, MFA required, tag values)

Example condition: "aws:RequestedRegion": "us-east-1" blocks use of the role outside one region.

Explicit Deny always wins over Allow — useful for guardrails.

Detective Controls

Detective controls find problems after they happen so you can respond.

  • GuardDuty — threat detection service; watches network, DNS, and audit logs for suspicious activity
  • Security Hub — central dashboard that aggregates findings from GuardDuty and others, scores them against standards like CIS
  • AWS Config — records the state of every resource and lets you write rules like "no S3 bucket may be public"

VPC — Network Anatomy

  • VPC — your isolated virtual network in AWS, spanning multiple AZs
  • Subnet — a slice of the VPC's IP range, tied to one AZ
  • Public subnet — has a route to an Internet Gateway; resources here can be reached from the internet
  • Private subnet — no direct internet path; outbound goes through a NAT Gateway
  • Pattern: web/load-balancer in public subnets, app and database in private subnets

Network Protection Controls

  • Security groups — per-instance firewalls, stateful (return traffic is automatic). Default-deny.
  • NACLs (Network Access Control Lists) — subnet-level firewalls, stateless. Rarely needed beyond defaults.
  • WAF (Web Application Firewall) — blocks L7 attacks: SQL injection, XSS, bot traffic
  • Shield — protection against DDoS attacks (Shield Standard is free; Advanced is paid)
  • Network Firewall — deep-inspection firewall for egress filtering
  • PrivateLink / VPC endpoints — reach AWS services without traversing the public internet

Data Protection

  • At rest (stored on disk): use KMS (Key Management Service) to manage encryption keys — most AWS services encrypt by default
  • In transit (moving across the network): use TLS (Transport Layer Security — the "s" in https)
  • Classify data — not every data set needs the same controls
  • Backups must be encrypted and tested
  • Secrets Manager / Parameter Store — never put passwords or API keys in code or env files
  • Rotate keys and secrets on a schedule, automatically if possible

Incident Response Readiness

  • Pre-provision a forensic account isolated from production
  • Pre-built break-glass roles responders can assume in an emergency
  • Know how to: revoke a session, rotate a key, quarantine an instance, invalidate a token
  • Document contact lists and escalation paths
  • Practice with tabletop exercises and live simulations

An Over-Permissive Policy

{
  "Effect": "Allow",
  "Action": "*",
  "Resource": "*"
}

An IAM policy is a JSON document. "*" means "everything." Three red flags on five lines. We'll find more in the lab.
