Pillar 3

Reliability

Recover from failure and handle changing demand.

AWS Geography — Regions & AZs

  • AWS is divided into Regions (e.g., us-east-1 in Virginia) — geographically separate
  • Each Region has multiple AZs (Availability Zones) — physically separate data centers within the Region, connected by low-latency links
  • An AZ failure is common enough to plan for; a Region failure is rare but catastrophic

Foundations

  • Know your service quotas (AWS limits on how many resources you can create) and raise them early
  • Plan network topology & IP address ranges up front
  • Reliability is built in before the first deploy, not after the first outage

Stateless vs. Stateful

Core concept for scaling and recovery.

  • Stateless component — holds no data between requests; any instance can handle any request (e.g., a typical web server)
  • Stateful component — remembers something between requests (sessions, data, pending work) — e.g., a database
  • Stateless components scale easily: add more copies, route traffic across them
  • Stateful components are the hard part of reliability — that's where backups, replicas, and failover matter
  • A good pattern: push state into a managed service (RDS, DynamoDB) so the rest of your app can stay stateless

Failure Management

  • Fault isolation — divide the system so one failure can't take down everything (e.g., multi-AZ)
  • Idempotent operations — safe to retry (running it twice has the same effect as once)
  • Backups — automated, encrypted, restores tested (untested backups often don't work)
  • Auto Scaling — AWS automatically adds/removes servers based on metrics (CPU, queue depth, etc.)

Blast Radius

When something fails, how much of the system goes with it?

  • A small blast radius means one failure affects one thing — one AZ, one shard, one customer
  • A big blast radius means one failure takes down the whole system — everyone is down at once
  • Techniques to shrink it: multi-AZ, multi-region, cells/shards, canary deploys, feature flags
  • Every architecture decision changes blast radius — know in which direction

RTO & RPO

  • RTO (Recovery Time Objective) — how long can the system be down?
  • RPO (Recovery Point Objective) — how much data can you afford to lose?

Example: "RTO 1 hour, RPO 5 minutes" = back online within an hour, losing at most 5 minutes of data.

DR Pattern Ladder

DR = Disaster Recovery, the plan for surviving a big failure.

  • Backup & restore — take backups, restore when needed. Cheapest, slowest.
  • Pilot light — keep a minimal core of the system running in another region
  • Warm standby — full copy running small in another region, scaled up on failover
  • Active/active — both regions serve traffic all the time. Most expensive, fastest.

Pick by business need, not aspiration.
