Pillar 4

Performance Efficiency

Match resources to the workload so it runs fast enough without waste.

Latency vs. Throughput

Two different "fast" — and they sometimes trade off.

  • Latency = how long a single request takes (milliseconds)
  • Throughput = how many requests you can handle per second
  • A batch pipeline optimizes for throughput; an API optimizes for latency
  • Caching often improves both; adding replicas often improves throughput but not single-request latency

Scaling: Horizontal vs. Vertical

  • Vertical scaling — make the server bigger (more CPU, more RAM). Simple, but has a ceiling and a restart.
  • Horizontal scaling — add more servers. Needs a stateless design and a load balancer, but has no ceiling.
  • AWS Auto Scaling is horizontal by default
  • Databases historically scaled vertically; modern services like DynamoDB and Aurora scale horizontally

Compute Options

  • EC2 — traditional virtual machines; you manage the OS
  • Containers — package code + dependencies, run on:
    • ECS (Elastic Container Service) — AWS's own container orchestrator
    • EKS (Elastic Kubernetes Service) — managed Kubernetes
    • Fargate — run containers without managing servers at all
  • Lambda — serverless functions; you provide code, AWS runs it on demand

Storage Options

  • S3 — object storage for files, backups, web assets; cheap and virtually unlimited
  • EBS (Elastic Block Store) — hard-disk-style volumes attached to an EC2 instance
  • EFS (Elastic File System) — shared file system mountable by many servers at once
  • FSx — managed Windows / Lustre / NetApp file systems for specialized needs

Database Options

  • RDS / Aurora — managed relational databases (MySQL, PostgreSQL, etc.)
  • DynamoDB — managed NoSQL key-value database; scales to huge volumes
  • ElastiCache — managed in-memory cache (Redis, Memcached)
  • OpenSearch — managed search and analytics engine

Measure, Don't Guess

  • Load test before launch and after big changes
  • Performance Insights — RDS/Aurora tool that shows the slowest queries
  • X-Ray — find the slow piece of a distributed request
  • Track p95 / p99 latency — "95% (or 99%) of requests finish in under this time"
    Averages hide slow outliers; percentiles show what users actually feel.

Trade-Offs

  • Caching — store frequently-read data in fast memory (ElastiCache, DAX for DynamoDB)
  • Read replicas — copies of a database that serve read queries, offloading the primary
  • CDN (Content Delivery Network) — push content to servers near users. AWS's CDN is CloudFront.
  • Each adds complexity — don't add until you've measured the need
