Module 8

Practical Application & Wrap-Up

Running a Well-Architected Review

  • Open the Well-Architected Tool in the AWS Console (the web interface at console.aws.amazon.com)
  • Define the workload — name, owner, region(s), environment
  • Walk pillar by pillar, answering honestly — not aspirationally
  • Save a milestone — a snapshot of your answers — as your baseline

Reading the Output

  • Each finding is classified as High Risk, Medium Risk, or None
  • Each finding links to an improvement plan with AWS documentation and services
  • Lenses — pillar add-ons for specific workload types (Serverless, SaaS, Machine Learning)

What a Finding Looks Like

Example from a real review:

  • Pillar: Reliability
  • Question: "How do you back up data?"
  • Answer given: nightly snapshots, never tested
  • Risk: High
  • Improvement plan links to: AWS Backup, cross-region copy, game day guidance
  • Team's action item: "Schedule quarterly restore drill, owner: SRE lead, due Q2"

The value isn't the score — it's the conversation that produces the action item.

Prioritizing Findings

  • High-risk Security and Reliability items first — these can hurt customers
  • Then quick wins (low effort, medium risk)
  • Defer items with low impact and high cost — but document why

Building a Remediation Roadmap

  • Group related findings into themes (e.g., "IAM hygiene", "DR readiness", "Cost tagging")
  • Each theme gets an owner and a target milestone
  • Sequence by dependencies — don't start "re-architect for multi-region" before "fix backups"
  • Budget ~20% of engineering capacity for framework-driven work
  • Track in the normal backlog alongside features — not in a parallel spreadsheet nobody reads

Maturity Over Time

  • First review: expect many High-risk findings. That's normal, not shameful.
  • After quarter 1: Highs resolved, Mediums remain
  • After quarter 2: process becomes routine; design-time checklists prevent new debt
  • Long-term: the framework becomes invisible — it's just how you build
  • Milestones in the tool let you prove progress to leadership

Integrating into Your SDLC

SDLC (Software Development Life Cycle) = how your team goes from an idea to running code.

  • Lightweight review at design time — catch issues before code is written
  • Automated guardrails: Config rules, SCPs (Service Control Policies — account-wide "never allow" rules), CI/CD policy checks
  • Full review at major milestones and before launches
  • Track findings in your normal backlog — not a separate spreadsheet
  • Goal: continuous, not annual
